Start with explicit boundaries

A durable reader starts with a simple promise: the article you save today should still be readable tomorrow, even when the network is gone. The implementation details matter because every hidden dependency becomes a future failure mode.

The capture layer should be narrow. It fetches through a single client, extracts readable structure through a testable parser, and hands the result to a repository that already owns persistence. The user interface should not know whether the parser is SwiftSoup, WebKit, or something else.

Prefer predictable fallbacks

When a page cannot be extracted, the correct result is an error that the user can act on. Saving a blank document is worse than failing because it turns a transient capture problem into permanent library data.

This is also why tests use fixtures. A fixture can represent a real page shape while remaining deterministic, fast, and independent from the public internet.
